OMAR
Open Modular Access to Records
Proposal: OMAR as a scaffold for collaborative, agile development of the Health Summarization project

Summary

OMAR is a clinician‑developed, AI‑connected VA clinical assistant prototype that aligns closely with the prior health‑summarization MVP scope. We propose using OMAR as a scaffold to accelerate development, validation, and deployment of a production‑ready summarization tool - one whose integration into a clinical workflow suite will likely increase user adoption. We propose shifting to a collaborative, agile development model focused on OMAR, with a core team comprised of the current full-time development team and 1-2 part-time clinicians, and pulling in contributors from the VA Clinical Innovations Group.

OMAR Philosophy

The Clinician Problem

  • Great, specific tools (CDS, JLV, search) that lack integration.
  • Information overload and alert fatigue.
  • Use of AI carries significant risk, especially in clinical decision-making.
  • Commercial tools fail to understand diverse workflows within the VA.

Goals

  • Clinical flow design based on the elegance of CPRS; feels natural for CPRS users.
  • AI tools feel natural, traceable and always include human‑in‑the‑loop. Design metaphor is a diligent medical student.
  • Support clinical decision making by surfacing data in the right place and time, clearly avoiding clinical recommendations.
  • Build EHR-agnostic and modular for the future. Support clinician‑developers in extending the platform.

How OMAR aligns with the prior MVP scope

  • Pre‑encounter chart review: Hey OMAR Q&A pulls both structured data and unstructured notes together to answer user questions, whether a general health summary, a condition-specific overview or a narrow query.
  • Structured data inclusion: Dot‑phrases and endpoints for vitals, labs, meds, problems.
  • Human‑in‑the‑loop: Drafts and answers require clinician review; sources shown inline.
  • Trust & safety foundations: Every assertion is traceable with in-line linked citations, encouraging immediate verification.

Clinical perspective: why use OMAR as the base

  • Built by clinicians: designed around CPRS‑adjacent workflows.
  • Beyond summarization: ambient scribing, structured data views, fast search, RAG Q&A, and after‑visit instructions.

Collaborative Development Model

  • Natural fit: OMAR was designed from the start to be modular. Features can be added independently, and feedback will be frequent. Agile-style development fits the bill.
  • Open‑source agile: public repo (internal or external per policy), issue triage, small PRs, weekly sprints, transparent roadmap.
  • Current core dev team as maintainers: product + engineers + UX steward architecture, security and deployment, and validation coordination.
  • Current OMAR clinician-developers: steward overall design and feature development; provide rapid informed feedback early in processes.
  • VA Clinical Innovations Group and others: contribute features, prompts, and validation via structured pull requests and requests for comments with code review and safety checks.
  • Communication & norms: weekly stand‑ups, async updates, contribution guidelines, code of conduct, and onboarding sessions.

Phases

Phase 0 — Team Development, Open‑Source Onboarding & Coordination

  • Team building: current MVP dev team and OMAR clinicians learn from each other, create shared vision and trust.
  • Relationships: find and listen to IT stakeholders, including CPRS/CDS/Lighthouse teams, and clinical stakeholders.
  • Communication & norms: stand‑ups, decision logs, contribution guidelines, code of conduct.
  • Onboarding & teaching: short workshops for VA Clinician Innovators and other contributors; dev environment setup; PR walkthroughs.
  • Governance: maintainer roles, process for clinical feature requests and safety changes.
  • Infrastructure for collaboration: repo, issues/labels, project board, templates (PR, issue, RFC), documentation site.

Phase 1 — Secure Foundations

  • Core team focus: product solidity, security, deployment, and validation coordination.
  • User authentication and secure server deployment
  • Transition fully from socket to API
  • Request needed RPCs for CDS context
  • Data‑safety hardening to enable early user testing

Phase 2 — Early Use & Tuning

  • Early user testing with rapid feedback
  • Initial performance tuning
  • Prompt refinement for granular clinical needs

Phase 3 — Build & Validate in Parallel Parallel

3A Technical Stabilization Parallel

  • Read endpoints: API access (VistA, HIE, Oracle Cerner, My HealtheVet); extract patient/user/facility; standardize and transform.
  • Analytics/AI layer: improve RAG; consider knowledge graphs and complementary methods.
  • Security & performance: robust error logging, PHI‑safe telemetry.
  • Write endpoints: notes, reminders/tasks and encounters only.

3B Clinical UX & Workflows Parallel

  • Workspace and UI flows focused on simplicity, clinical flow, and flexibility.
  • Clinician tabs: inpatient rounds, outpatient prep, social work, etc.
  • Alert management, secure messaging, and VistA write‑backs (unsigned notes/tasks).
  • Frontend: connect to backend; safe, efficient workspace (patient switch, archiving, speech‑to‑text); reusable UI tools (graphs, tables). SDK for clinician developer to make custom tabs for submission.

3C Validation & Safety Parallel

  • De‑identified charts; standardized questions and rubrics for completeness, correctness, and relevance.
  • Broaden across care settings and professions.

Phase 4 — Pilot Rollout

Limited rollout to pilot sites with monitoring and outcome metrics.

Validation framework

ComponentMethod
CompletenessStandard scenarios with known info requirements
CorrectnessExpert review vs. sources
RelevanceClinician utility ratings
SafetyCritical miss/incorrect detection
Qualitative interviewsSemi‑structured interviews; defer to UI/UX team member
  • Create robust de‑identified patient charts (MIMIC‑style).
  • Standardize question sets for summarization and notes.
  • Conduct qualitative interviews.

Expected outcomes

  • Secure, scalable deployment with stable endpoints and SDK for clinician‑developers.
  • High clinician adoption, satisfaction and ongoing use among test users.
  • Validated summarization quality and safety across settings and professions.
  • Limited rollout with measurable adoption and outcome metrics.
